<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| request -> server page | Listing id in route params is untrusted and must be filtered by authenticated user. |
| server page -> database | Score rows are read from `opportunity_scores` and must be scoped to `user.id`. |
| server page -> client component | UI-safe score entries cross from server-rendered page into client component props. |
| browser -> server action | Compute/recompute remains server-side through `scoreListingAction`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Tampering | route id + selected strategy | mitigate | Route keeps listing query filtered by `.eq('user_id', user.id)` and score mutation goes through `scoreListingAction`, which validates `STRATEGY_SLUGS`. |
| T-18-02 | Information Disclosure | `getScoreHistory` page query | mitigate | Page calls `getScoreHistory(supabase, user.id, id)` only after `getUser()`; no client API fetch can request another user's scores. |
| T-18-03 | Tampering | client score props | mitigate | Page passes persisted rows through `scoreRowToCardEntry`; browser cannot submit score totals or category values. |
| T-18-04 | Information Disclosure | raw scoring JSON | mitigate | Page passes UI-safe card entries and card renders capped signal/risk labels; no raw JSON row is printed. |
</threat_model>
