<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> server action | Browser sends listing id and strategy slug to `scoreListingAction`; server must validate and compute score. |
| server page -> database | Authenticated server page reads listing and score rows filtered by `user_id`. |
| persisted scoring JSON -> UI | Stored `breakdown`, `signals`, and `risks` JSON crosses into display helpers. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Tampering | strategy slug helpers/tests | mitigate | Helper tests require strategy values to flow from `STRATEGY_SLUGS`; later server action validation remains the authority for forged slugs. |
| T-18-02 | Information Disclosure | `scoreRowToCardEntry` | mitigate | Adapter maps only UI-required score fields and `score_version`; tests must not expose `user_id` or raw row JSON in card entries. |
| T-18-03 | Tampering | client score values | mitigate | Helpers do not compute scores from client input; score values come from persisted rows or server action return shape only. |
| T-18-04 | Information Disclosure | scoring JSON display | mitigate | Helper output supports capped display rows and does not provide raw JSON dump behavior. |
</threat_model>
