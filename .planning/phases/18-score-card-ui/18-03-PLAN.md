<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> server action | User interaction triggers scoring with listing id and strategy slug. |
| server action -> browser | Returned `ScoringActionState` may include score, message, or errors. |
| persisted score row -> visual card | Saved score entries become broker-facing explainability UI. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Tampering | `OpportunityScoreCard` strategy action | mitigate | Card sends only `listingId` and `selectedStrategy`; selector restricts UI values and `scoreListingAction` validates `STRATEGY_SLUGS` server-side. |
| T-18-02 | Information Disclosure | saved score entries | mitigate | Card receives `initialScores` from the authenticated server page only; it must not fetch arbitrary listing/user score data from the client. |
| T-18-03 | Tampering | total/category scores | mitigate | Card never computes score values; it renders server-provided `ScoreResult` from persisted rows or action return. |
| T-18-04 | Information Disclosure | signals/risks JSON | mitigate | Card renders at most three signal labels and three risk labels, no raw JSON dumps, no `user_id`, and no full database row display. |
</threat_model>
